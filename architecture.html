Architecture Blueprint

FootballGPT Orchestration Engine

An advanced Multi-Agent architecture combining Semantic Memory Retrieval, Two-Stage Tool-Calling, and Real-Time Memory Deduplication.

A3
Pre-Process (Blocking)

Agent 3: Retriever

Runs immediately when a user message is received. It is responsible for searching the long-term vector store for historical context.

  • Generates embeddings for user query
  • Searches ChromaDB vector collection
  • Retrieves top matches (similarity > 0.5)
  • Injects matching contexts dynamically into Agent 1's system prompt
A1
Main Stream (User-Facing)

Agent 1: Converser

The core conversational engine of FootballGPT. It generates the reply to the user and triggers external API actions.

  • Fitted with systemic football knowledge
  • Equipped with API-Football capabilities
  • Executes complex, 2-stage tool-calling
  • Maintains localized thread history
A2
Post-Process (Async)

Agent 2: Memorizer

Runs asynchronously in a background thread pool executor so the user does not wait. Stretches memory longevity.

  • Extracts core facts in structured JSON
  • Calculates cosine similarity to prevent repeats
  • Deduplicates memories (threshold > 0.95)
  • Persists unique facts in ChromaDB

Deduplication & Cosine Space

When the memory extraction agent (Agent 2) finds a new fact, it is vital to avoid storing redundant memory blocks (e.g., repeating "User supports Arsenal" multiple times).

distance = CosineDistance(emb_new, emb_old)
similarity = 1.0 - (distance / 2.0)
if similarity > 0.95: SKIP_STORE()

Key Storage Attributes:

  • Unique ID: Stored via UUIDv4
  • Embedding: Multi-dimensional array representing semantic layout
  • Document: Cleaned text descriptor of the fact
  • Metadata: ISO timestamp to manage time-sensitive recall

2-Stage Tool Execution Loop

Instead of replying blindly, Agent 1 is loaded with structural tool definition schemas. Emojis and other configurations are fine-tuned for a fast, responsive interface.

1

LLM Decides Action

The model receives the conversation history. It decides if the user's question requires real-time data (e.g. current standings) and returns a structural tool call payload instead of final text.

2

Interception & Execution

The Flask application intercepts the tool call, matches the function name against local handlers, runs the request against the external api-sports endpoint, and formats the output string.

3

Data Feed & Final Answer

The formatted output is injected as a tool role message in the conversation thread. The model runs a second completion cycle to build a natural conversational response utilizing the retrieved metrics.

Available API-Football Handlers:

get_standings(), get_fixtures(), get_live_scores(), get_player_stats(), get_top_scorers()
